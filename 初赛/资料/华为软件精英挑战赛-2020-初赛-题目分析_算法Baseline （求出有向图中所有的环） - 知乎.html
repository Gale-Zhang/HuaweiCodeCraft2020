题意：给定一个有向图，求出图中所有长度在[3,7]之间的环。

输入：格式为[IDU,IDV,~]的边表，ID为32位无符号整数（当然题目说明了，小于2^31，所以int32就好），边最多28W条，不重复，结点平均度数小于10。环中同一个ID不可以重复出现（若大环包括小环，则需要分开统计），环的个数不大于300W。

输出：环的个数，按照1.环长度；2.环的数字字典序输出所有环。

举个例子：1,2,3的全连接图有两个环

初步思路（Updated on 200407）：

  • 要避免搜索出的路径重复，需要制定一个环的标准型，以题目为例，就是最小的ID在环表示的首位
  • 题目要求最小的结点在每个路径的第一位，故对ID排序后，重映射所有结点序号为[0,n)，在搜索时，如果直接对某个结点的出边从小到大搜索，最后得到的答案自然是排好序的（虽然排序一遍答案花掉的时间也不多）
  • 用拓扑排序删去一部分点（入度为0，或出度为0的点可以删去，直到没有满足条件的点）
  • 搜索，剪枝（深度在3-7之间、每次访问的结点ID都应该大于起点的ID），这里有很大的优化空间
  • 关于换行应该是CRLF（\r\n）还是LF（\n），官方表示都没有关系

初步优化思路（Updated on 200407）：

大佬们总是在炫耀运行时间，让我们这种菜鸡情何以堪

在暴力膜的基础上，该如何进行优化？

通过观察不难发现，在比较稠密的图上，长度为N的环的数量相比长度为N-1的环的数量是指数增长的（和结点的平均度数有关）。

如果能通过只搜索到第6层，而非第7层就得到所有的解，就能够将运行的时间有效的缩减到原先的1/6左右。

我们定义如下的数据结构：

vector<unordered_map<int,vector<int>>> P2;

使用P2[i][j][k]来表示结点i到达结点j，中间经过结点k的路径详情，如果k不在我们已经搜索过的结点列表中，并且j是起点，那么i-k-j就是符合要求的路径的一部分。

具体来说，就是提前做深度为2的搜索（保存最后一层结点的入边），在第6层时直接根据现有结果进行判断，不进入第七层。

Update on 200409：

今日话题：如何判断一个图是稠密图？以及如何针对数据特点进行策略组合。

一般来说，结点平均度数随着结点数量增长的是稠密图，如果结点的度数稳定在某一个范围（例如0~20），与结点数量无关，那么就是稀疏图（例如赛题）。

虽然我们嘴上常说，稠密图使用邻接矩阵存储，稀疏图使用邻接表存储，但是像这种结点平均度数小于10的，怎么也不是稠密图（用邻接矩阵存储内存直接爆炸好嘛）。如果一定要使用邻接矩阵来表示比赛的数据，可以考虑离散化以后使用可变长向量进行存储（vector套娃，参考baseline）。

在官方给定的限制条件下，我们可以很容易的造出一些数据，用来评测算法在边界条件下的表现，例如：

  • 28W边，56W不同结点（鉴于所有大于28W的结点都可以预处理到小于28W，这是唯一一组结点数大于28W的数据）
  • 28W边，28W不同结点，长度在3-7的单向环
  • 28W边，14W不同结点，长度在3-7的双向环
  • 28W边，28W不同结点，组成一个大环

所以我们说的稠密与稀疏是相对的，边的数量和结点数量决定了结点的平均度数，图的稠密程度就主要与结点度的均值和标准差有关（当然，这只是一个比较Naive的指标）。

以上述数据集为例，我们可以统计其结点的入度、出度、度数和的均值、标准差如下：

  • 【边数、点数、答案、耗时（v0408）、度数均值、标准差】
  • E28W N30000 A1004812 12s 9.3 3.0
  • E28W N28500 A2861665 13s 9.8 3.1
  • E28W N25700 A2896262 18s 10.9 3.3
  • E28W N25000 A3512444 20s 11.2 3.3
  • E28W N59989 A2755223 27s 4.7 6.9

在随机生成的数据上，度数的均值其实就是边数除以点数，而度数的标准差就是度数均值的开方左右。但是其中有一组数据非常的辣眼睛：270W的那组数据之所以会耗时很大，是因为其中部分结点的度数非常高（以某个结点为起点的环数约70W），表现为度数的标准差很大。

在此基础上，我们可以得到一个用于应对稠密图和稀疏图的组合策略。当度数均值较大，或者度数的标准差较大时，以稠密图求解的方案来应对，反之，则按照稀疏图的方案来应对。

虽然官方表示所有环的个数不会大于300W，但是对于每个环数的大小没有明确的说明。

公布昨晚的抽奖结果（真的勇士，敢于直面WA和RE），在线所有环的数量不会大于下列数值：

int ans3[3*500000];
int ans4[4*500000];
int ans5[5*1000000];
int ans6[6*2000000];
int ans7[7*3000000];

为了避免针对不同的数组写一大堆判断代码，不妨把所有数组的开头指针整合起来：

int *ans[]={ans3,ans4,ans5,ans6,ans7};
//e.g. 
//for path length 6 ansCnt6 50
//we can set ans[3][50][5] by calling ans[3][50*6+5]=x

计算一下内存占用，可以从100MB缩小到40MB左右。

此外，有大佬测试了最大的出度为不大于50，入度不大于255（据说入度也是不大于50的），目前可以采用的图存储方式（vector套娃类型）为：

int G[280000][50]; //G[u][v]
int GInv[280000][255]; //G[v][u]
